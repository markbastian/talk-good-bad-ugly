Threading State

Representing Application State

  • Use a very small number of values to represent application state.
    • Small number is preferably 1
  • Maintain a persistent reference to your values using concurrency primitives
    • Atoms, agents, refs
  • Round-trip state values
    • Primitive ->
    • Self similar functions
    • swap! or alter. Not reset.
    • -> Primitive

Interacting with State

  • Producing Changes
    • UI layer (Swing, etc.) will call swap!, reset!, alter, etc.
    • A single set-validator! can be used for input validation across ALL modifiers.
  • Consuming Changes
    • Any number of add-watches can be added to your reference
    • When watches are triggered, make changes to output system
    • Note: Watch for sync issues (SwingUtilities/invokeLater)
  • Concerns of I/O independent of value management

Thread-Ready State

  • Write your API such that X' = f(X)
  • X, X' are persistent data structures
  • X, X' are stucturally similar

Separate Concerns

  • Lots of small functions
  • Dissassociate functions from other concerns
    • Concurrency primitives
    • Event handlers
    • Other UI
